## MyGarage UI specs: Hamburger sidebar + header search

This file is written so an AI agent can replicate the same UX in another application.

---

## A) Header hamburger sidebar (pop-out drawer) — replication spec

### A1) What it is
- **Left-side drawer** that slides in/out from the hamburger in the header.
- Used for **category navigation** (“Find parts”) with a **sidebar-local search** that switches between:
  - **Browse mode**: hierarchical collapsible tree (`<details>/<summary>`).
  - **Search/list mode**: flat list of matches with breadcrumbs.
- Supports **hover-open (temporary)** and **pinned-open (sticky)** behavior.

### A2) Visual design
- **Placement**: `fixed` left, below the header:
  - Mobile: `top-14` and `h-[calc(100vh-3.5rem)]`
  - Desktop: `md:top-16` and `md:h-[calc(100vh-4rem)]`
- **Width**: `w-[min(100vw-1rem,340px)] max-w-[86vw]`
- **Layering**: sidebar `z-50`, overlay `z-40`
- **Surface**: `bg-card`, `border-r border-border`, `shadow-xl`, `overflow-hidden`
- **Animation**: translate X with `transition-transform duration-200`
  - Open: `translate-x-0`
  - Closed: `-translate-x-full`
  - Also gates interaction: open → `pointer-events-auto`, closed → `pointer-events-none`

### A3) Open/close behavior (state machine)
Implemented in `components/header.tsx` and passed into `components/additems-sidebar.tsx`.

- **State**:
  - `pinned: boolean`
  - `hoverOpen: boolean`
  - `open = pinned || hoverOpen`
- **Hover-open** (when not pinned):
  - `onMouseEnter` (hamburger or sidebar) → `hoverOpen = true`
  - `onMouseLeave` → schedule close in **150ms** (prevents flicker)
- **Pinning**:
  - Clicking hamburger toggles `pinned`
  - Pin on → sidebar opens
  - Pin off → sidebar closes
- **Closing**:
  - Click overlay backdrop → close
  - Press `Escape` (while open) → close
  - If pinned, the sidebar shows a small **“Close”** button in its own header → close
  - Clicking category links closes the drawer **only when not pinned**

### A4) Overlay backdrop
- Rendered only when `open`.
- `fixed inset-x-0 top-14 md:top-16 bottom-0 z-40 bg-black/20 backdrop-blur-sm`
- Click handler closes the sidebar.

### A5) Sidebar content behavior (inside drawer)
Implemented in `components/additems-sidebar.tsx`.

- **Data loading**:
  - Fetches `GET /api/additems` with `{ cache: 'no-store' }`
  - Expects JSON: `{ items: Array<{ title: string; children?: same[] }> }`
  - `items === null` → skeleton loading UI
  - `items.length === 0` → “No categories loaded.”
- **Browse mode** (sidebar query empty):
  - Renders full tree with `<details>` expansion
  - Chevron is a “›” character that rotates on open (`group-open:rotate-90`)
  - Children are indented with a left border (`border-l border-border/50`)
- **Search/list mode** (sidebar query non-empty):
  - Flattens the tree into rows containing:
    - `title`
    - `breadcrumb` (ancestors joined with ` › `)
  - Filters rows by title or breadcrumb containing the query
  - Sorts matches by relevance (exact > startsWith > includes; breadcrumb is weaker) then alphabetically
  - Empty state includes a helpful hint (“Try a shorter word…”)

## B) Header search bar (products & services search) — replication spec

This is the **search input in the header**, separate from the sidebar’s “Find parts” search.
Implemented in `components/header.tsx`.

### B1) Core idea: URL-driven search with debounce (“Amazon-like”)
- The app treats the query as the URL param **`q`** on the homepage:
  - With query: navigates to `/?q=<encoded>`
  - Empty query: navigates to `/`
- There are two pieces of state:
  - `urlQ`: the current query derived from the browser URL (`window.location.search`)
  - `searchValue`: what the user is currently typing in the input
- On mount, `urlQ` initializes from the current URL.

### B2) Keeping input in sync with browser navigation
- A `popstate` event listener updates `urlQ` when the user uses back/forward.
- Whenever `urlQ` changes, the component **updates the input** by setting `searchValue = urlQ`.

### B3) Debounced navigation while typing
- When `searchValue` changes, a `useEffect` sets a **250ms timer**.
- After 250ms:
  - If `searchValue.trim()` equals `urlQ`, it does nothing.
  - Otherwise it calls `applySearch(searchValue)` which performs `router.replace(...)`:
    - non-empty → `router.replace("/?q=...")` and `setUrlQ(next)`
    - empty → `router.replace("/")` and `setUrlQ("")`

### B4) Submit behavior (Enter key / form submit)
- The input sits inside a `<form onSubmit={...}>`.
- Submitting:
  - prevents default
  - calls `applySearch(searchValue, { closeSidebar: true })`
  - This is a convenience so searching also collapses the hamburger drawer if it happens to be open.

### B5) Escape key behavior inside the search input
- On `Escape` key down in the input:
  - `setSearchValue('')`
  - `applySearch('', { closeSidebar: true })` (clears URL query and closes sidebar)
  - `closeSuggestions()` (hides the suggestions panel)

### B6) Live suggestions panel (desktop + mobile)
The header search can show a suggestions dropdown/panel once focused.

- **Visibility state**: `suggestionsVisible: boolean`
  - `onFocus` → shows suggestions immediately (`showSuggestionsNow()`)
  - `onBlur` → schedules close after **120ms**
    - This delay allows the user to click a suggestion without the input losing focus and closing the panel first.
- **Fetch trigger rules**:
  - Suggestions only fetch when `suggestionsVisible === true`
  - Query must be at least **2 characters** (`q.length < 2` clears suggestions and errors)
- **Fetch implementation**:
  - Debounced by **200ms** after changes to `searchValue`
  - Uses `AbortController` to cancel in-flight requests when the query changes/unmounts
  - Endpoint: `GET /api/search/suggestions?q=<encoded>`
  - Handles:
    - `suggestionsLoading` state while fetching
    - `suggestionsError` (shows “No suggestions” on errors other than abort)
- **Result model** (as used by the UI):
  - `categories: { name, image, count, headline }[]`
  - `products: { id, name, description, price, compareAtPrice, image, category, brand }[]`
  - Optional: `serviceCategories: { categoryId, categoryTitle, emoji, count, headline, topServiceName }[]`
  - Optional: `services: { id, name, categoryId, categoryTitle }[]`

### B7) Desktop suggestions UI behavior
- Rendered as an **absolute dropdown** under the search input when:
  - `suggestionsVisible && searchValue.trim().length >= 2`
- Uses `onMouseDown(e.preventDefault())` on the dropdown wrapper so clicking inside doesn’t blur the input before the click runs.
- Clicking any suggestion link:
  - navigates (category/product/service routes are built via helper functions)
  - calls `closeSuggestions()`
  - calls `closeSidebar()` (so the hamburger drawer closes too)

### B8) Mobile suggestions UI behavior (fixed sheet below the search field)
- On mobile, suggestions render as a **fixed, full-width panel** (`fixed inset-x-0`) with high z-index `z-[60]`.
- The panel’s `top` and `maxHeight` are computed dynamically so it sits right under the search field and fits the viewport:
  - A `useLayoutEffect` measures the search field (`getBoundingClientRect().bottom + 6`)
  - Uses `visualViewport.height` when available for better keyboard/viewport behavior
  - Updates on `resize`, `scroll` (including capture), and visualViewport resize/scroll
- Panel is scrollable (`overflow-y-auto`) and uses `overscroll-y-contain`.
- Like desktop, it prevents blur-on-click with `onMouseDown(e.preventDefault())`.

### B9) Route builders used by suggestions
- Categories:
  - If category is empty or “all” → `/`
  - Else → `/category/products/<encoded category>`
- Services:
  - Service category page → `/buyer/services?sc=<categoryId>&quick=1`
  - Specific service quick link → `/buyer/services?sc=<categoryId>&ss=<serviceName>&quick=1`
